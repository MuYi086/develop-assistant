# 门店订货与补货通用链路

## 分层地图

| 层 | 责任 |
| --- | --- |
| API | 分页、可选商品、汇总、创建/更新、详情、状态操作、导出 |
| Vuex（可选） | 跨组件共享的记录、详情或稳定缓存 |
| 页面容器 | 筛选、路由、可编辑副本、流程编排和最终 DTO |
| 子组件 | 分类商品、基础信息、汇总展示和行级事件 |
| 路由 | 列表、创建、编辑、详情及权限 metadata |

先按宿主结构映射这些角色。简单页面可直接调用 API；不要为了匹配模板强行创建 Vuex module。

## 典型 API 能力

| 动作角色 | 语义 |
| --- | --- |
| pageOrders | 订货/补货单分页 |
| listGoods | 分类与可选商品 |
| summarize | 服务端计算总数量、金额、税费或优惠 |
| create/update | 创建或编辑保存 |
| getEditorData | 获取门店、已有数量和编辑版本 |
| getDetail | 获取权威详情 |
| transition | 取消、完成、审核或备注等状态动作 |
| exportDetail | 导出当前详情 |

这些是语义别名，不是固定函数名。实际 method、字段与响应层级使用宿主接口契约。

## 创建/编辑数据流

```text
路由进入
  -> 获取门店/已有订货数据
  -> 获取可选分类商品
  -> 按稳定商品 ID 合并已有数量
  -> 深拷贝为页面本地编辑模型
  -> 用户修改数量
  -> 服务端汇总或本地精确汇总
  -> 提取有效商品并构建 DTO
  -> 创建或更新
  -> 返回列表或详情
```

- 不按数组下标合并回显；使用商品 ID、SKU ID 或后端提供的稳定组合键。
- 分类下所有商品数量为 0 时不生成脏提交项。
- 提取函数负责去重、数值归一化并只保留 `quantity > 0`，汇总与提交复用同一结果。
- 编辑提交携带服务端版本号或更新时间时，不得被回显清理逻辑移除。

## 异步汇总与竞态

数量连续变化可能产生乱序响应，采用：

1. 每次输入立即增加 revision，使旧响应失效；
2. 防抖后发送当前有效商品快照；
3. 响应只在 revision 与发送时一致时写入；
4. 提交前 flush 防抖并等待最新汇总；
5. 汇总失败时阻止提交或明确使用宿主允许的降级策略。

取消请求只能节省资源，不能替代 latest-wins，因为取消可能晚于响应。若汇总只服务当前页面，优先让 API/action 返回数据，由页面决定接纳，不让旧响应直接 commit 全局汇总。

## 子组件与状态归属

- 父页面拥有可编辑商品树；子表格接收数据并 emit 最小变更。
- 不直接修改 prop、Vuex getter 或列表行对象。
- 只把多页面共享且刷新策略明确的数据放入 Vuex；临时输入与请求 revision 留在页面。
- action 返回 Promise/数据，让页面协调提交；mutation 只做同步更新。

## 详情与状态操作

- 状态标签与按钮权限统一来自宿主枚举/状态机。
- 取消、完成、审核和备注即使共用 endpoint，也要构建不同的动作 DTO。
- 状态操作成功后重新请求详情，不手工拼接局部状态。
- 导出同时验证查询身份、Content-Type、服务端文件名、错误 JSON 和浏览器 URL 回收。

## 回归重点

- 回显合并不受分类或商品排序变化影响。
- 汇总响应乱序时 UI 与最终 DTO 始终对应最后输入。
- 全部数量为 0 时阻止提交，合法金额 0 不被当成缺失。
- 重复点击、接口失败和路由返回不会遗留 loading 或脏 store。
- 状态变化后按钮、详情、列表和导出使用同一权威结果。
