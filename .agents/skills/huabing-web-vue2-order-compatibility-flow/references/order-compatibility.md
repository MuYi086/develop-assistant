# 订单兼容通用矩阵

## 必须显式考虑的维度

| 维度 | 示例角色 | 主要影响 |
| --- | --- | --- |
| 订单版本/能力 | legacy、current | 路由、字段和页面能力 |
| 订单来源 | 自有端、POS、第三方渠道 | 退款渠道、提示和可操作性 |
| 商品结构 | 普通商品、组合/套餐 SKU | 表格、价格和售后明细 |
| 履约阶段 | 未开始、进行中、已完成 | 可退数量、库存与损耗 |
| 售后方式 | 仅退款、退货退款 | DTO、库存回滚和资金动作 |
| 价值类型 | 金额、积分、成长值、优惠权益 | 展示、校验和回收字段 |

矩阵中的值是语义角色，不是固定枚举。先读取宿主 API 文档、类型或 options，再建立映射。

## 路由与身份

```text
入口记录 -> 取得稳定 orderId/orderNo/version/capabilities
         -> 统一路由选择器
         -> 目标详情路由
```

- 不要把支付单 ID、退款单 ID 或第三方流水号当成订单 ID。
- 支付详情、退款详情、订单列表和关联业务入口复用同一选择语义。
- 若旧/新详情并存，集中维护版本到路由的映射；未知版本应给出可诊断结果，而不是静默跳错页。
- 参数名与 path 由宿主项目决定，参考 Skill 不提供可复制的固定路由。

## 数据归一化

在页面边界将不同版本响应转换为稳定 view model：

```js
{
  identity,
  source,
  capabilities,
  goodsGroups,
  fulfillmentSteps,
  priceSummary,
  afterSaleSummary
}
```

原始枚举保留在适配层，展示组件只消费明确字段。缺失数组归一化为空数组；`0` 与缺失值分别处理。

## 普通商品与套餐 SKU 售后

- 普通商品按商品行计算最大可退数量。
- 套餐按父商品分组，在组内选择 SKU 数量；提交时只保留 `refundNum > 0` 的可退项。
- 未履约与进行中项目可能对应不同库存、损耗或退款规则，不能只按 UI 分组后合并 payload。
- 空选择阻止提交；提交前再次比较用户选择与服务端最新可退信息。
- 操作成功后重新查询详情和售后能力，不能只在前端扣减。

字段 `refundNum` 仅为语义示例；实际 DTO 使用宿主接口字段。

## POS 与第三方渠道

- 使用稳定来源枚举或 capabilities，不用显示文案识别渠道。
- 若只能原渠道退款，禁用不允许的金额/方式，并在提交层再次阻止。
- “仅回滚库存、不在本系统退款”与“本系统发起资金退款”是不同动作，payload 不得携带另一动作的明细。
- 第三方可能不接收本系统售后明细；按能力适配，不复制某个渠道的专属字段。
- 新增支付方式时同步检查 options、支付记录、详情、导出和售后能力。

## 一致性与竞态

- 售后弹窗打开后订单状态可能变化；提交使用版本号、幂等键或提交前再校验。
- 成功后同步刷新主详情、操作记录、履约进度和可退信息。
- 旧请求响应不能覆盖操作后的新状态；使用取消、请求 ID 或 revision/latest-wins。
- 金额使用精确小数/最小货币单位，避免浮点直接相加。

## 回归样例

- legacy 普通订单从支付记录进入正确详情。
- current 套餐订单从退款记录进入并保持订单身份。
- POS/第三方订单无本系统退款能力时不生成资金明细。
- 套餐部分未履约、部分进行中时，只提交允许且选择数量大于 0 的 SKU。
- 售后完成后价格、记录、步骤、按钮和非现金权益同时刷新。
