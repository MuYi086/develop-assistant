# 话饼 Web Vue2 订单兼容流

## 开始前

1. 读取 `references/order-compatibility.md`。
2. 明确改动覆盖的订单版本、来源、商品结构、履约阶段、价值类型和售后动作。
3. 检查详情 API、页面组件、售后弹窗，以及所有可能跳入订单详情的入口。

## 实施流程

1. 建立兼容矩阵，决定每个差异落在路由选择、数据归一化、computed、展示组件、DTO 构造或 API 选择中的哪一层。
2. 路由入口使用稳定的后端版本/能力字段选择目标详情，并显式传递订单 ID、订单号等宿主所需参数。
3. 主详情页获取和归一化数据；商品、步骤、履约、价格与售后组件各自处理单一展示或操作。
4. 普通商品与组合/套餐分开构造售后明细；套餐按 SKU 过滤用户实际选择且可退款的数量。
5. POS 或第三方来源按能力与渠道规则限制前端操作；提示、可编辑字段和最终 payload 必须一致。
6. 操作成功后重新获取权威详情及相关记录，保证进度、可退数量、价格和售后状态同步。

## 约束

- 版本与来源判断集中为可测试的谓词或适配器，不在模板中散落魔法数字和渠道字符串。
- 后端枚举值不由前端重定义；显示文案通过宿主 options/解析层映射。
- 金额、积分、成长值、优惠权益和数量分别校验，不能用 truthy 判断吞掉合法的 `0`。
- 售后数组只提交用户选择且服务端允许的项；UI 隐藏不等于 payload 自动安全。
- 跳转修复要搜索订单列表、支付记录、退款记录和其他关联入口。
- 不调用真实支付、退款或外部 POS 系统做普通前端验证。

## 验证

1. 覆盖旧/新版本、内部/POS/第三方来源、普通商品、套餐和各履约阶段。
2. 覆盖仅退款、退货退款、SKU 部分数量、无可退项和并发状态变化。
3. 从所有入口核对目标路由、订单身份参数和返回行为。
4. 核对金额、优惠、售后记录、非现金权益回收和步骤状态。
5. 运行宿主项目定向 lint/测试并审查 payload diff；构建前确认无部署、推送、通知或发布副作用。
