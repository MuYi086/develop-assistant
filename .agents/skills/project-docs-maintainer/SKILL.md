# 项目文档维护器

维护准确、实用、视觉清晰且有仓库证据支撑的 GitHub 项目文档。除非项目已有明确语言规范，面向用户的文档和任务报告默认使用中文。

## 工作方式

1. 识别一个或多个模式：`init`（初始化或重构文档）、`sync`（仅同步受代码变更影响的文档）、`audit`（评审文档并修复已确认的问题）。
2. 先阅读仓库约束，再从最小必要范围的源码、配置、测试和现有文档收集证据。不得依据项目名称或惯例推断事实。
3. 按下表只读取当前任务需要的参考文件；不要默认加载全部参考文件。
4. 有证据支撑时直接修改所需文档；保留有价值的人工内容，不改动无关文档。

| 场景 | 读取的参考文件 |
| --- | --- |
| 任意文档任务 | `references/core-workflow.md` |
| 创建、重写或美化 README | `references/readme-design.md` |
| 根据代码变更同步文档 | `references/change-scope.md` |
| 涉及命令、链接、媒体、徽章、图表或最终检查 | `references/verification.md` |
| 文档评审或发布就绪检查 | `references/audit.md` |
| 输出面向用户的完成报告 | `references/reporting.md` |

## 基本约束

- 将可执行代码、配置、测试和已验证的公开资源视为证据；无法确认的冲突应明确记录，不得虚构结论。
- 不得声称未经验证的功能、命令、兼容性、性能、路线图状态、截图、徽章、下载量、用户数量或评价。
- 优先做聚焦的增量更新。只有当一个长期主题会让 README 过长时，才创建补充文档。
- 有仓库修改权限时，不得用“建议大纲”代替实际编辑。仅要求评审时，只报告问题并修复已确认的缺陷。
- 命令、路径、标识符、包名和协议名必须保持可执行；中文文档中必要的英文术语首次出现时补充中文说明。

## 完成任务

执行 `references/verification.md` 中与改动相关的检查，查看最终差异，并按 `references/reporting.md` 输出报告。清楚区分已验证事实、未验证项、缺失资源和剩余文档债务。
