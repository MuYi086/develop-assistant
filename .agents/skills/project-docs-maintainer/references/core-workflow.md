# 核心工作流

每次调用都应阅读本文件。它规定证据标准，并帮助选择范围最小且安全的文档改动。

## 基于证据检查仓库

编辑前先阅读适用的仓库说明。随后仅检查能够确认当前文档事实的文件：

- 现有 `README*`（项目说明文档）、`docs/`（文档目录）、`CONTRIBUTING.md`、`SECURITY.md`、变更记录、发布说明和许可证。
- 包清单与锁文件、构建和测试配置、CI（持续集成）配置、环境变量示例、部署与容器文件。
- 当任务涉及相关内容时，检查应用入口、路由、API（应用程序接口）或 CLI（命令行界面）定义、数据结构、测试和公开资源。
- 进行 `sync`（同步维护）时，检查当前 `git status`（工作区状态）以及相关的已暂存和未暂存差异。

除非它们是唯一可靠来源，否则不要读取依赖目录、构建产物、缓存、第三方代码和自动生成文件。源码与配置通常比现有文案更可靠；两者冲突时，也不得武断认定其中一方正确。

## 选择模式

### `init`（初始化或重构文档）

适用于文档不完整、过时、模板化、难以浏览，或无法反映成熟项目现状的场景。

1. 确认项目用途、目标读者、已实现能力和实际入门路径。
2. 评估现有文档中值得保留的已验证内容、错误、重复和缺失入口。
3. 按项目类型设计信息架构；阅读 `readme-design.md`。
4. 重建或重组 README，只新增确有必要的专题文档。
5. 每条可供外部用户执行或依赖的事实，都必须先验证再写入。

`init` 是基于项目实际情况的重新设计，而不是套用通用 README 模板。

### `sync`（同步维护文档）

适用于功能开发完成后，或用户要求保持文档与代码一致时。

1. 检查 `git status`、已暂存变更和未暂存变更。
2. 识别面向用户、运维人员、接口使用者或贡献者的行为变化。
3. 阅读 `change-scope.md`，只更新受本次变更影响的文档。
4. 检查新增、删除或弃用的命令、配置、接口、路由、部署步骤和示例。
5. 若代码差异没有公开文档影响，在完成报告中说明判断理由。

小功能变更不得引发对无关 README 的整体改写。

### `audit`（文档评审）

适用于发布、开源、重大合并、对外推广前，或怀疑文档已偏离实际实现时。

1. 将高价值文档声明与仓库证据逐一对比。
2. 阅读 `audit.md`，按严重程度记录已确认问题。
3. 除非用户只要求评审，否则直接修复已确认且安全的缺陷。
4. 明确记录冲突和无法验证的事实，不得用看似合理的文案填补空缺。

模式可以组合使用；例如评审可能发现需要进行一次小范围 `sync`。

## 保护已有文档

保留已验证的项目历史、品牌决策、法律文本、致谢、维护者说明、设计背景、有效的故障排查记录和兼容性说明。仅在内容不准确、重复、只是占位、缺乏依据、已废弃，或明确处于重写范围内时修改或删除。

## 判断包管理器

锁文件是初始依据。若存在多个锁文件，应结合项目配置、CI、贡献指南和近期维护约定判断，而非任意选择。

| 锁文件 | 默认包管理器 |
| --- | --- |
| `pnpm-lock.yaml` | pnpm |
| `yarn.lock` | Yarn |
| `package-lock.json` | npm |
| `bun.lock` 或 `bun.lockb` | Bun |
| `uv.lock` | uv |
| `poetry.lock` | Poetry |
| `Pipfile.lock` | Pipenv |
| `Cargo.lock` | Cargo |
| `go.sum` | Go 模块 |
