# README（项目说明文档）设计与写作

创建、重组或明显优化 README 与文档信息架构时，读取本文件。

## 信息架构

只保留能够回答真实读者问题的章节，不要机械地添加完整模板。

1. 项目标识和一句话价值说明
2. 项目状态，以及少量有价值且可验证的徽章
3. 当前截图、演示效果或基于事实的架构示意图（确有助于理解时）
4. 核心能力和代表性使用场景
5. 快速开始和必要配置
6. 使用示例
7. 有助于贡献者理解的架构或目录结构
8. 按实际需要补充开发、部署、路线图、贡献和许可证信息

首屏应让访问者快速理解项目是什么、服务谁、解决什么问题、当前状态如何、实际效果或行为是什么，以及如何开始。完整接口说明、部署流程和较长的操作材料应在能降低 README 阅读成本时拆到已链接的专题文档。

## 区分读者

面向普通用户，优先说明项目作用、安装、启动、配置、使用方式、常见恢复路径、维护状态和平台限制。面向贡献者，只记录已验证的开发上下文：架构、目录边界、本地环境、开发与测试命令、代码与提交约定、发布流程及已知技术债务。除非运行项目确有必要，不要把实现细节混入用户入门流程。

## 按项目类型突出重点

| 项目类型 | 优先展示 |
| --- | --- |
| Web（网页）应用 | 当前截图、核心用户流程、功能概览、在线演示、启动路径、环境变量、架构、部署 |
| CLI（命令行界面）项目 | 安装方式、最小可运行命令、参数、输入输出示例、配置、自动化使用方式 |
| SDK（软件开发工具包）或开发库 | 安装命令、最小代码示例、核心接口、兼容性、类型、错误处理、版本和迁移策略 |
| AI（人工智能）或模型项目 | 工作流、输入输出、硬件要求、推理方式、模型来源、许可限制、可复现的性能条件、限制和示例 |
| 后端或基础设施项目 | 架构、依赖服务、本地开发、配置、接口、迁移、监控、部署、故障排查 |

## 补充文档

只有当专题内容长期存在且足够充实、继续放在 README 会妨碍浏览时，才创建专题文档。常见位置包括 `docs/getting-started.md`、`docs/configuration.md`、`docs/architecture.md`、`docs/deployment.md`、`docs/troubleshooting.md`、`docs/api.md`、`docs/cli.md`、`docs/development.md` 和 `docs/migration.md`；只有仓库有实际内容依据时，才创建 `CONTRIBUTING.md`、`SECURITY.md` 或 `CHANGELOG.md`。每份新增文档都必须从有意义的导航入口链接到。

不得为了让仓库看起来完整而创建内容空洞的文件。

## 视觉与文案质量

视觉元素只应降低理解成本：有效截图、已有项目标志、少量必要徽章、简明的 Mermaid（文本化图表语法）图表、对比表格、示例或可折叠的高级说明。避免徽章墙、过量表情符号、仅为装饰而使用的 HTML（超文本标记语言）居中布局、无必要目录，以及用装饰掩盖信息不足。

文案应具体且有证据，说明问题、读者、已实现能力、可执行路径、范围、限制和状态。除非仓库证据能够支持，否则避免“强大”“无缝”“行业领先”“生产就绪”等泛化宣传语，也不要使用通用欢迎语或宽泛受众承诺等人工智能模板套话。

公开文案默认使用中文。可执行英文名称必须保持原样；除项目另有语言要求外，必要英文技术术语首次出现时补充中文说明。
