# 文档评审

执行 `audit`（文档评审），或在发布、开源前检查文档时，读取本文件。

## 评审重点

检查 README（项目说明文档）是否准确描述当前项目，安装命令是否与包管理器一致，环境变量和配置是否完整，链接与资源是否有效，截图是否反映当前版本，示例是否仍使用当前 API（应用程序接口），许可证是否正确。还要检查弃用功能是否仍被宣传、路线图状态是否真实、首屏是否快速传达价值和状态，以及用户与贡献者是否有可行的入门路径。

同时识别缺乏依据的宣传语、模板化套话、与代码冲突的文档、已经偏离实际的架构或目录说明，以及不符合仓库语言规范的内容。

除非用户只要求评审，否则修复已确认缺陷。无法验证的声明应如实记录，不得编造替代内容。

## 严重程度与问题格式

默认用中文分组输出问题：

- **阻断问题**：可能阻止安装、启动、部署或正确使用，例如缺少必填变量、无效命令、危险指引、不存在的功能或错误许可证。
- **重要问题**：明显损害理解、维护或可信度，例如错误架构图、不完整配置、过时截图、错误路线图状态或缺少贡献者测试流程。
- **一般问题**：风险较低的质量问题，例如标题不一致、表格难读、内容重复、术语不统一或 README 首屏过于拥挤。
- **优化建议**：能提升清晰度、专业性或可维护性，但不是必要修复的建议。

每条问题都应包含位置、当前问题、影响、修改建议和判断依据。不要只写“建议优化”等空泛结论。
