# 验证与证据保护

文档包含命令、资源、图表、徽章、路线图，或即将完成编辑时，读取本文件。

## 命令

记录命令前，确认包管理器、命令或脚本本身、执行目录、所需运行时和环境变量、依赖服务、预期结果以及操作系统限制。安全且可行时实际运行命令。无法运行时，不得把推测写成事实；应在完成报告中说明未验证状态，而不是在面向用户的文案中暴露内部猜测。

## 截图、媒体和徽章

仅在截图来自当前版本、路径可在 GitHub 正常渲染、画面清晰、不包含敏感信息或测试账号，并且确实有助于理解项目时使用。不得虚构截图，也不得拿其他项目的图片充当占位。若缺少有价值的截图，应向维护者请求或在报告中列为缺失资源。

徽章必须有可验证来源和实际价值，例如构建、测试、许可证、版本、覆盖率、包管理器或发布状态。不得添加纯装饰技术栈墙，也不得在没有依据时声称下载量、用户数量、排名或兼容性。

## Mermaid 图表与路线图声明

Mermaid（文本化图表语法）适合表达有证据支撑的架构、数据流、请求流、部署拓扑、模块关系、状态流或核心流程。图表应保持可读；复杂图表需要简短说明；不得把规划中或猜测的组件画成已实现。

路线图状态应区分已完成、开发中、计划中、评估中、暂缓和已取消。问题单、设计稿、未合并分支、占位代码、实验和不可达实现都不能称为已完成功能。

## 最终检查

完成前查看最终差异，并按本次改动检查：

- 标题层级、Markdown（轻量级标记语言）渲染风险、相对链接、图片路径、导航链接和新增文档是否可被发现。
- 命令、配置、公开接口和功能声明是否与仓库依据一致。
- 是否存在虚构事实、过时描述，或把规划功能写成已发布功能。
- README 首屏是否清晰、是否符合语言规范、必要英文术语是否在首次出现时有中文说明。
- 是否无意义地重写了无关的人工文档。

明确说明未执行的检查及原因。不得声称已运行构建、测试、链接检查器或命令，除非确实执行过。
