# UniApp Vue 2 订单售后流

## 实时资格优先

- 进入取消/退款前向服务端重新查询资格、时限、责任类型、可退项和金额。
- 页面缓存只用于首屏展示，不能作为最终可取消/可退依据。
- 把服务端资格归一为 `not-allowed`、`direct`、`rule-required`、`manual-review` 等领域状态，再由路由/弹窗决定 UI。
- 未知状态安全降级并提示刷新，不默认执行退款或取消。

## 参数归一

- 在入口一次性归一主订单、交易单、子单、联合单、版本、支付状态和来源页。
- query 中的 Boolean/Number 显式转换；后续逻辑只使用规范字段。
- 旧字段在 adapter 边界转换，不让同一字段在不同页面代表不同 ID。
- 多版本详情入口使用 versioned-route 技能，避免列表与详情复制不同矩阵。

## 取消流程

1. 获取最新资格和取消成本/规则。
2. 需要规则确认时展示 dialog，倒计时或责任时限以服务端时间为准。
3. 构造明确 command，并携带幂等键。
4. 提交期间锁定按钮；成功后刷新订单并按来源决定返回，失败保持上下文并恢复交互。
5. 联合单、子单和未支付订单分别按后端契约处理，不靠前端拆单猜测。

## 退款与异常履约

- 可退商品、数量、金额、履约项和原因来自同一服务端快照。
- 用户选择使用 draft，提交前重新校验数量、金额和资格。
- 部分退款金额由服务端计算；前端合计只用于展示和早期校验。
- 0 元、无可退项、重复提交、审核中和退款失败都有独立状态。
- 成功后保存最小结果摘要或重新查询售后单，不依赖来源页临时 data。

## 状态与清理

- 建模 `idle/checking/eligible/submitting/success/reviewing/failed`，不使用多个互相矛盾的 Boolean。
- 页面重进、切换订单或用户注销时重置售后草稿和提交锁。
- socket/轮询只发布状态事件，页面收到后重新查询权威售后状态。
- 错误展示兼容统一 error contract，并保留服务端可读消息。

## 验证

- 不允许、直接取消、需规则、人工审核和资格在提交前变化。
- 普通单、联合单、子单、未支付单、旧/新 ID。
- 全额/部分/0 元退款、数量边界、重复提交和网络重试。
- 列表与详情入口结果一致，完成后的返回栈和状态刷新正确。
