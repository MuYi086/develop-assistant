# 订单同步最佳实践

> 生成时间：2026-06-26  
> 输入文档：`当前版本订单同步机制流程.md`、`研究订单同步.md`  
> 适用范围：POS 本地订单、云端订单、在线/离线支付、退款、库存、沽清、烹饪工单的双向同步治理

## 一句话结论

当前订单同步最大的问题不是“同步链路不够多”，而是 **同一笔订单被 POS、云端、支付回调、库存逻辑、烹饪设备共同修改，但系统还没有把字段权威、业务操作、补偿重试、冲突处理显式建模**。

最佳实践不是继续扩大“完整订单快照互推”的能力，而是把订单拆成多个权威域：订单基础信息、支付、退款、库存、沽清、烹饪、交付。每个权威域要有明确的负责人、版本、幂等键、补偿机制、审计日志和冲突处理策略。

## 一、当前流程文档暴露出的核心问题

### 1. 完整聚合根覆盖风险过高

当前流程中，POS 推云端时发送 `OrderSyncCloudDTO(完整聚合根)`。云端存在订单且有变化时，会执行：

- `ACTION_OVERWRITE`
- `preserveCloudAuthoritativeFields()`
- `delete旧聚合根→save新数据`

这个设计的风险是：订单聚合根里混合了多个权威域，不能用一个覆盖策略处理。

订单里至少包含：

- POS 本地下单信息
- 支付字段
- 退款字段
- 商品退款明细
- 库存副作用
- 烹饪工单状态
- 出餐交付状态

如果云端只保护 20 个支付/退款字段，但其他字段仍通过删除旧聚合根再保存新聚合根处理，仍然可能出现丢字段、状态回退、明细丢失、设备状态被旧快照覆盖。

**建议改法：**

不要把 `ACTION_OVERWRITE` 理解为整单覆盖。应改为“按权威域合并”：

| 数据域 | 推荐处理 |
| --- | --- |
| 订单基础信息 | POS 建单场景下 POS 优先，云端后台订单云端优先 |
| 在线支付字段 | 云端/支付回调强权威 |
| 离线支付字段 | POS 强权威 |
| 在线退款结果 | 云端/支付回调强权威 |
| 退货商品动作 | POS 强权威，但要和退款操作号关联 |
| 烹饪工单状态 | POS/IOT/设备事件强权威，禁止旧快照回退 |
| 库存扣减/回滚 | 不靠订单覆盖，改用库存流水和 outbox |
| 沽清状态 | 商品售卖域独立权威，禁止被订单快照顺带覆盖 |

工程上建议把 `preserveCloudAuthoritativeFields()` 升级为 `mergeByAuthorityDomain(localSnapshot, cloudSnapshot, context)`，并输出合并决策日志。

### 2. “保护 20 个支付/退款字段”仍然不够

当前文档写到：如果 `isCloudOrderPaid() = true`，云端保护 20 个支付/退款字段，例如 `payType`、`amountPaid`、`refundStatus`。

这个判断粒度偏粗。

在线退款场景里，冲突不只发生在“订单是否已支付”这个层面。更关键的问题是：

- POS 是否发起了退款？
- 这次退款请求的业务操作号是什么？
- 云端收到的支付回调对应哪次退款？
- 退款金额、退款商品、退款渠道单号是否对应？
- 本地是否已经执行退货和库存回滚？
- 云端退款回调是否只是“退款受理”，还是“退款成功”？

如果只有字段保护，没有退款操作链路，系统仍然无法判断“这两个退款状态是不是同一次退款”。

**建议改法：**

将退款从订单快照中抽出来，建立独立 `RefundOperation`：

| 字段 | 说明 |
| --- | --- |
| `refundOperationId` | POS 和云端共同识别的一次退款业务操作 |
| `refundRequestId` | 请求支付渠道时使用的请求号 |
| `channelRefundId` | 支付渠道返回的退款单号 |
| `refundSchId` | 商品退款明细关联键 |
| `orderId` | 所属订单 |
| `source` | `POS` / `CLOUD_ADMIN` / `PAYMENT_CALLBACK` |
| `stage` | `LOCAL_RETURNED` / `REQUESTED` / `CALLBACK_SUCCESS` / `CALLBACK_FAILED` / `COMPENSATING` / `MANUAL_REQUIRED` |
| `idempotencyKey` | 防止重复退款和重复回调消费 |
| `localVersion` / `cloudVersion` | 操作发生时的本地/云端版本 |
| `createdAt` / `updatedAt` | 审计时间 |

订单主表上的 `refundStatus` 只作为投影字段，由退款操作流推导。

### 3. `delete旧聚合根→save新数据` 不适合高风险子域

删除旧聚合根再保存新数据，对于普通订单备注或展示字段可以接受，但对支付、退款、库存、烹饪工单不安全。

这些子域都有外部事实来源：

- 支付渠道回调
- 退款渠道回调
- 库存流水
- 烹饪设备回调

一旦通过新快照替换旧聚合根，很容易把外部事实写没，或者让状态从高阶段退回低阶段。

**建议改法：**

高风险子域禁止 delete-save 式覆盖：

- 支付和退款：只允许通过支付事件、退款事件、回调事件推进。
- 库存：只允许通过库存流水和库存操作事件调整。
- 烹饪工单：只允许通过设备事件或 POS 本地工单事件推进。
- 冲突字段：不覆盖，进入冲突记录。

### 4. `BeanUtils.copyProperties(local→cloud)` 这类通用复制很危险

当前拉取云端订单应用到 POS 时，文档里写到：

```text
BeanUtils.copyProperties(local→cloud)
保留POS业务字段
applyCloudPaymentAuthority()覆盖支付字段
mergeRefundDetails()按refundSchId合并
```

问题不在 `BeanUtils` 这个工具本身，而在“先大范围复制，再补丁式覆盖”的思路。

当订单对象字段越来越多时，新增字段如果没有加入保护清单，默认就可能被复制覆盖。支付、退款、库存、烹饪这种高风险字段，不能依赖开发者每次都记得加排除字段。

**建议改法：**

采用白名单合并，而不是黑名单保护：

```text
mergeOrderSnapshot:
  mergeBaseInfo()
  mergePaymentByAuthority()
  mergeRefundByOperation()
  mergeCookingByMonotonicState()
  mergeDeliveryByAuthority()
  ignoreInventorySnapshot()
  ignoreSaleOutSnapshot()
```

每个 merge 方法只处理自己负责的数据域，并输出：

- 输入版本
- 被改字段
- 采用本地值还是云端值
- 采用规则 ID
- 是否产生冲突

### 5. 冷却期内 `ACKNOWLEDGE_CLOUD_VERSION` 语义不清

当前流程中，`SITE_POS + CONFLICT(冷却期内)` 会返回 `ACKNOWLEDGE_CLOUD_VERSION`。

这个设计可能有合理背景：避免短时间内反复冲突，让系统等一等。但它也有风险：如果“确认云端版本”被理解为“这个冲突已经处理”，可能掩盖真实冲突。

**建议改法：**

冷却期内不要只 ACK，应记录一个“延迟决策”：

| 字段 | 说明 |
| --- | --- |
| `orderId` | 订单 |
| `localVersion` | 本地版本 |
| `cloudVersion` | 云端版本 |
| `cooldownUntil` | 冷却结束时间 |
| `blockedDomains` | 被延迟处理的数据域 |
| `reason` | 延迟原因 |
| `nextAction` | 冷却后重拉、自动合并、人工处理 |

冷却期结束后必须有明确动作，不能只依赖后续定时任务偶然修复。

### 6. 冲突只标记，不足以解决问题

当前流程有 `MARK_CONFLICT→记录冲突`，但文档没有描述冲突记录包含什么，也没有描述如何处理。

仅有 `syncStatus = CONFLICT` 不够。运营和研发排查时需要看到冲突详情。

**建议改法：**

建立 `OrderSyncConflict`：

| 字段 | 说明 |
| --- | --- |
| `conflictId` | 冲突 ID |
| `orderId` | 订单 ID |
| `domain` | `PAYMENT` / `REFUND` / `INVENTORY` / `COOKING` / `DELIVERY` / `BASE` |
| `localVersion` | 本地版本 |
| `cloudVersion` | 云端版本 |
| `localValue` | 本地值快照 |
| `cloudValue` | 云端值快照 |
| `fieldDiff` | 字段差异 |
| `operationIds` | 相关操作号 |
| `riskLevel` | `P0` / `P1` / `P2` |
| `resolutionMode` | 自动合并、人工处理、补偿任务 |
| `resolutionResult` | 处理结果 |
| `createdAt` / `resolvedAt` | 时间 |

### 7. 库存扣减失败只 `log.warn`，缺少补偿和对账

当前流程明确写到：

- 云端库存扣减失败：`log.warn 不回滚`
- `无自动重试/对账任务`

这不是小问题。订单保存成功但库存扣减失败，会导致订单和库存不一致。短期看订单不丢，长期看库存会越来越不可信。

**建议改法：**

库存必须从订单同步里拆出来，改为库存流水和补偿任务：

```text
OrderCreated
  -> InventoryDeductRequested(stockOperationId)
  -> InventoryDeductSucceeded / InventoryDeductFailed
  -> failed 时进入库存异常队列
  -> 定时对账订单明细、库存流水、商品库存
```

库存扣减失败可以不回滚订单，但必须生成：

- 库存异常记录
- 可重试任务
- 对账任务
- 人工处理入口

### 8. 库存回滚通过事务后 HTTP 通知云端，存在丢事件风险

当前流程写到：

- POS 本地事务内执行 `inventoryRollback()`
- 事务后调用 `postCloudStockGoodsBatch()` 通知云端回增
- 通知失败时无补偿重试/对账任务

这是典型的“本地数据库已提交，但外部消息发送失败”问题。

**建议改法：**

采用 Transactional Outbox：

1. 本地事务内同时写入库存回滚结果和 `inventory_outbox`。
2. 事务提交后，由后台 worker 读取 outbox。
3. worker 调用云端库存接口。
4. 云端用 `stockOperationId` 幂等消费。
5. 成功后标记 outbox 已处理。
6. 多次失败进入死信/人工队列。

这样本地库存回滚和“必须通知云端”这个事实不会分离。

### 9. 退款明细只按 `refundSchId` 合并不够

当前流程里，`mergeRefundDetails(local, cloud)` 按 `refundSchId` 合并，同键云端覆盖，本地独有保留。

这比整单覆盖安全，但还不够覆盖复杂退款：

- 同一商品可能多次部分退款。
- 一次业务退款可能拆成多个渠道退款单。
- POS 本地退货和云端支付回调可能不是同一粒度。
- `refundSchId` 如果复用或缺失，会合并错。

**建议改法：**

退款明细合并键至少包含：

```text
refundOperationId + refundSchId + goodsId + skuId + channelRefundId
```

合并时同时校验：

- 商品数量
- 商品退款金额
- 订单退款总金额
- 渠道退款总金额
- 本地退货数量
- 库存回滚数量

金额和数量不平时，不自动覆盖，进入冲突队列。

### 10. 定时任务全量推送 pending 订单，容易放大乱序和重复

当前流程每 20 秒查询 `syncStatus != SYNCED`，并“覆盖烹饪工单/库存回退/退款/出餐等所有变更”。

兜底任务是必要的，但如果每次都推完整快照，会放大两个问题：

- 旧快照晚到，把新状态覆盖掉。
- 同一订单多次 pending，被多个任务重复推送。

**建议改法：**

兜底任务不应该直接表达为“推完整订单快照”，而应该表达为“补发未确认操作”：

```text
SELECT * FROM order_outbox
WHERE status IN ('NEW', 'FAILED_RETRYABLE')
ORDER BY orderId, sequence
```

同一 `orderId` 内按 `sequence` 保序，同一操作通过 `idempotencyKey` 保证可重复发送。

### 11. 沽清没有独立权威模型

任务描述特别提到“如果此时进行了沽清操作”。源流程文档没有明确沽清在订单同步里的权威归属。

沽清不是订单字段，它属于商品售卖域。它可能由门店操作、云端后台、库存规则共同触发。如果订单同步快照顺带覆盖沽清状态，风险很高。

**建议改法：**

沽清独立建模：

| 字段 | 说明 |
| --- | --- |
| `saleOutOperationId` | 沽清操作号 |
| `goodsId` / `skuId` | 商品 |
| `scope` | 门店、渠道、时间段 |
| `source` | POS、云端后台、库存规则 |
| `version` | 沽清版本 |
| `effectiveAt` / `expiredAt` | 生效范围 |
| `reason` | 原因 |

订单同步只能引用沽清结果，不能覆盖沽清权威数据。

## 二、推荐的订单同步设计原则

### 原则 1：订单快照用于展示，业务事实来自操作日志

订单快照适合做查询和展示，不适合作为所有业务真相。

真正的业务事实应该来自：

- `OrderCreated`
- `PaymentConfirmed`
- `RefundRequested`
- `RefundCallbackReceived`
- `GoodsReturned`
- `InventoryDeducted`
- `InventoryRolledBack`
- `SaleOutChanged`
- `CookingOrderSent`
- `MealDelivered`

快照可以由这些操作投影出来。短期不需要全量 Event Sourcing，但支付、退款、库存、沽清、烹饪这些高风险子域应先操作化。

### 原则 2：字段权威必须显式化

不要把“谁说了算”写散在 if/else 里。建议维护一张权威矩阵。

| 数据域 | POS 是否可写 | 云端是否可写 | 第三方回调是否可写 | 默认权威 | 冲突处理 |
| --- | --- | --- | --- | --- | --- |
| 订单基础信息 | 是 | 视订单来源 | 否 | 建单端 | 版本较新但不得覆盖高风险域 |
| 离线支付 | 是 | 否 | 否 | POS | POS 覆盖云端 |
| 在线支付 | 发起/展示 | 是 | 是 | 支付回调/云端 | POS 不得覆盖成功回调 |
| POS 退货动作 | 是 | 可展示 | 否 | POS | 和退款操作号关联 |
| 在线退款结果 | 发起/展示 | 是 | 是 | 支付回调/云端 | 以回调为资金事实 |
| 退款明细 | 是 | 是 | 是 | 按操作号和渠道单号合并 | 金额不平进冲突 |
| 库存扣减 | 是 | 是 | 否 | 库存流水 | 走补偿和对账 |
| 库存回滚 | 是 | 是 | 否 | 库存流水 | outbox 重试 |
| 沽清 | 是 | 是 | 可由库存规则触发 | 商品售卖域 | 独立版本冲突 |
| 烹饪工单 | 是 | 展示/下发 | 设备回调 | 设备事件/POS | 状态只能前进 |
| 出餐交付 | 是 | 可展示 | 设备回调 | POS/设备事件 | 状态只能前进 |

### 原则 3：每个跨端操作都要有幂等键

所有可能重试的操作都必须幂等：

- POS 推订单
- 云端拉订单
- 退款请求
- 退款回调消费
- 库存扣减
- 库存回滚
- 烹饪工单发送回调
- 出餐完成回调

推荐键：

```text
idempotencyKey = businessType + orderId + operationId + sequence
```

示例：

```text
REFUND:order_123:refund_op_456:1
STOCK_ROLLBACK:order_123:stock_op_789:1
COOK_CALLBACK:order_123:cook_event_222:1
```

幂等表至少记录：

- `idempotencyKey`
- `requestHash`
- `resultHash`
- `status`
- `firstSeenAt`
- `lastSeenAt`

同一 key 但参数不同，应拒绝并告警。

### 原则 4：跨系统事务用 Saga 和补偿，不追求分布式强事务

退款、库存、烹饪都跨越多个系统，不能指望一个数据库事务覆盖全部。

推荐使用 Saga 思路：

```text
本地事务成功
  -> 写操作日志
  -> 写 outbox
  -> 异步通知下一方
  -> 下一方幂等处理
  -> 失败后重试
  -> 多次失败后补偿或人工处理
```

在线退款建议使用编排式 Saga，因为它涉及 POS、云端、支付渠道、库存和订单状态，不适合完全靠各端事件自由编舞。

### 原则 5：高风险冲突要保留证据，不要自动覆盖

涉及钱、库存、退货商品、沽清、烹饪设备状态的冲突，不能为了“最终一致”直接覆盖。

推荐策略：

- 低风险字段自动合并，例如备注、展示字段。
- 中风险字段按权威矩阵合并，例如出餐状态只能前进。
- 高风险字段保留冲突，进入冲突队列。

## 三、推荐目标架构

### 1. 保留现有双向同步，但改变同步载荷含义

现有链路可以保留：

- POS → Cloud 推送
- Cloud → MQTT → IOT-Agent → POS 拉取
- 20 秒/2 分钟/24 小时兜底

但同步载荷应从“完整订单快照为主”逐步改为“操作 + 快照校验”。

```text
OrderSyncEnvelope
  orderId
  source
  direction
  baseLocalVersion
  baseCloudVersion
  operations[]
  snapshotDigest
  correlationId
```

其中 `operations[]` 承载业务变化：

```text
RefundRequested
RefundCallbackReceived
InventoryRolledBack
CookingOrderSendSucceeded
MealDelivered
```

快照只用于辅助校验和展示，不作为高风险域的覆盖来源。

### 2. 本地写入流程

```text
用户/POS操作
  -> 开启本地事务
  -> 更新订单查询快照
  -> 追加 OrderOperation
  -> 写 order_outbox
  -> 提交事务
  -> worker 异步推云端
```

事务内必须同时完成：

- 本地业务状态更新
- 操作日志写入
- outbox 写入
- `syncStatus` 标记

不能先提交业务状态，再临时调用外部 HTTP。

### 3. 云端接收流程

```text
收到 POS 操作
  -> 校验 idempotencyKey
  -> 校验 baseVersion
  -> 按 authority domain 应用操作
  -> 生成 SyncDecisionLog
  -> 必要时生成 Conflict
  -> 返回 ack(operationId, appliedVersion, decision)
```

云端必须能区分：

- 已处理重复请求
- 旧版本请求
- 可自动合并
- 需人工处理
- 需补偿重试

### 4. 云端下发流程

```text
云端订单/支付/退款变化
  -> 追加云端操作日志
  -> 发布 ORDER_SYNC 通知
  -> POS 拉取云端操作
  -> POS 按本地 pending 状态和权威矩阵应用
  -> 写 SyncDecisionLog
```

POS 拉取云端数据时，不能默认应用整单快照。应先判断是否有本地 pending，再按数据域合并。

## 四、关键子域最佳实践

### 1. 在线退款

在线退款是最高风险链路，推荐流程：

```text
POS 发起退款
  -> 本地创建 RefundOperation(refundOperationId)
  -> 本地记录退货动作，必要时先冻结库存回滚
  -> outbox 推送 RefundRequested 到云端
  -> 云端幂等创建退款请求
  -> 云端请求支付渠道，使用 refundRequestId/idempotencyKey
  -> 支付渠道异步回调云端
  -> 云端按 channelRefundId 更新 RefundOperation
  -> 云端下发 RefundCallbackReceived 给 POS
  -> POS 应用资金结果，并关联本地退货/库存状态
```

关键规则：

- POS 可以记录“已发起退款”和“已退货”。
- POS 不能把在线退款标记为最终成功，除非收到云端/渠道确认。
- 云端不能因为回调成功就覆盖 POS 的退货商品动作。
- 退款成功和退货完成是两个事实，必须通过 `refundOperationId` 关联。

### 2. 离线退款

离线退款的资金事实在 POS 侧，但仍需同步到云端。

推荐流程：

```text
POS 离线退款
  -> RefundOperation(source=POS, payMode=OFFLINE)
  -> 本地退款/退货/库存回滚事务提交
  -> 写 refund outbox + inventory outbox
  -> 网络恢复后推云端
  -> 云端按 POS 权威应用离线退款结果
```

云端可校验金额、商品数量、订单状态，但不应把在线支付规则套到离线支付。

### 3. 库存扣减和回滚

库存必须有流水，不要只看订单快照。

推荐表：

```text
InventoryLedger
  stockOperationId
  orderId
  goodsId
  skuId
  operationType: DEDUCT | ROLLBACK | ADJUST | SALE_OUT
  quantity
  source
  idempotencyKey
  status
  createdAt
```

本地和云端各自维护库存流水，通过 `stockOperationId` 做幂等和对账。

### 4. 沽清

沽清独立于订单同步。

订单可以因为商品沽清拒绝新下单，但已经存在的订单退款/取消不应被沽清状态打断。退款导致库存回滚后，也不应该自动取消沽清，除非有明确商品规则。

推荐规则：

- 沽清状态独立版本。
- 订单同步不得覆盖沽清状态。
- 退款库存回滚只产生库存流水，不直接改沽清。
- 如果业务要求“退款后恢复可售”，也必须由库存/商品规则显式触发 `SaleOutChanged`。

### 5. 烹饪工单

烹饪工单状态来自 POS、IOT-Agent、设备回调，推荐使用单调状态机。

示例：

```text
CREATED -> SENDING -> SEND_SUCCESS -> QUEUEING -> COOKING -> FINISHED
                        |
                        -> SEND_FAILED -> RETRYING
```

规则：

- 旧快照不能把 `SEND_SUCCESS` 回退为 `SENDING`。
- 设备回调必须带 `deviceEventId` 幂等。
- 出餐完成必须可追踪到设备事件或 POS 操作。

## 五、同步状态设计建议

当前 `syncStatus` 能表达大方向，但不足以表达业务域冲突。建议拆成两层。

### 订单同步总状态

```text
SYNCED
LOCAL_PENDING
CLOUD_PENDING
CONFLICT
MANUAL_REQUIRED
COMPENSATING
FAILED_RETRYABLE
FAILED_FATAL
```

### 数据域同步状态

```text
OrderDomainSyncState
  orderId
  domain: BASE | PAYMENT | REFUND | INVENTORY | SALE_OUT | COOKING | DELIVERY
  localVersion
  cloudVersion
  status
  lastOperationId
  lastDecision
```

这样一笔订单可以出现：

- 基础信息已同步
- 支付已同步
- 退款冲突
- 库存补偿中
- 烹饪已同步

比一个全局 `syncStatus` 更可诊断。

## 六、同步决策日志

每次同步都要回答“为什么这样合并”。

推荐 `OrderSyncDecisionLog`：

| 字段 | 说明 |
| --- | --- |
| `decisionId` | 决策 ID |
| `orderId` | 订单 |
| `direction` | `POS_TO_CLOUD` / `CLOUD_TO_POS` |
| `domain` | 数据域 |
| `action` | `APPLY_LOCAL` / `APPLY_CLOUD` / `MERGE` / `KEEP_LOCAL_PENDING` / `MARK_CONFLICT` / `SKIP_DUPLICATE` |
| `localVersion` | 本地版本 |
| `cloudVersion` | 云端版本 |
| `operationId` | 业务操作号 |
| `authorityRule` | 命中的权威规则 |
| `fieldDiff` | 字段差异摘要 |
| `reason` | 决策原因 |
| `correlationId` | 链路追踪 ID |
| `createdAt` | 时间 |

没有这张日志，线上订单错乱后很难复盘。

## 七、补偿、重试和人工处理

### 自动重试

适合：

- HTTP 超时
- MQTT 丢消息
- 云端短暂不可用
- 支付渠道查询超时
- 库存通知失败

要求：

- 幂等键
- 最大重试次数
- 指数退避
- 可观测重试状态

### 补偿任务

适合：

- 本地库存已回滚，云端未回滚
- 云端退款成功，本地未收到
- 烹饪工单发送状态卡住
- 支付渠道回调成功但订单投影失败

补偿任务必须记录进度，失败后可恢复。

### 人工处理

适合：

- 金额不平
- 商品退款明细对不上
- 支付渠道成功但本地订单不存在
- 库存流水和订单明细不一致
- 冷却期后仍然冲突

人工处理不是失败，而是高风险业务的正常兜底。

## 八、落地优先级

### P0：先防止钱和订单错

1. 输出字段权威矩阵，明确支付、退款、库存、烹饪、沽清谁说了算。
2. 在线退款引入 `refundOperationId`、`refundRequestId`、`channelRefundId`、`idempotencyKey`。
3. 支付/退款字段禁止被 POS 旧快照覆盖。
4. `mergeRefundDetails` 从 `refundSchId` 单键合并升级为多键合并和金额校验。
5. 建立 `OrderSyncDecisionLog`，至少覆盖支付和退款。
6. 冲突记录必须包含字段差异和处理状态。

### P1：补齐库存和外部副作用

1. 库存扣减失败生成异常记录，不只 `log.warn`。
2. 库存回滚使用 outbox，不再只依赖事务后 HTTP。
3. 云端库存操作使用 `stockOperationId` 幂等。
4. 建立库存对账任务。
5. 沽清独立版本，订单同步不得覆盖。
6. 烹饪工单状态改为单调推进，旧快照不得回退。

### P2：降低完整快照同步权重

1. 推送载荷增加 `operations[]`。
2. 高风险域以操作为准，快照只做投影校验。
3. 同一订单 outbox 按 sequence 保序发送。
4. 定时任务从“推所有未同步订单快照”升级为“补发未确认操作”。
5. 冲突队列接入后台可视化处理。

## 九、测试场景清单

| 优先级 | 场景 | 关键断言 |
| --- | --- | --- |
| P0 | POS 在线退款发起后，云端先收到渠道退款成功回调 | POS 本地退货事实保留，云端资金结果生效，不重复退款 |
| P0 | POS 推旧快照到云端，云端已有支付成功 | 支付成功字段不被覆盖 |
| P0 | 同一退款请求重复发送 | 通过 `idempotencyKey` 返回同一结果，不产生第二笔退款 |
| P0 | 同一订单多商品部分退款 | 退款明细按操作号和明细键合并，金额不丢 |
| P0 | 退款金额本地和云端不一致 | 不自动覆盖，进入冲突队列 |
| P1 | 本地库存回滚成功，云端 HTTP 通知失败 | outbox 保留任务，重试后云端幂等回滚 |
| P1 | 云端库存扣减失败 | 订单保存不丢，但生成库存异常和对账任务 |
| P1 | 退款过程中商品被沽清 | 沽清状态不被订单快照覆盖 |
| P1 | 烹饪工单已 `SEND_SUCCESS`，拉到旧云端快照 `SENDING` | 本地状态不回退 |
| P1 | MQTT 丢失 | 兜底拉取能恢复，但不重复应用操作 |
| P2 | 冷却期内冲突 | 生成延迟决策记录，冷却后有明确后续动作 |
| P2 | 冷却期外冲突 | 记录字段差异，进入人工处理 |
| P2 | 定时任务并发推同一订单 | 同一 operation 只应用一次，同一订单内保持顺序 |

## 十、对当前流程的建议改写方向

当前 `当前版本订单同步机制流程.md` 已经很好地描述了链路，但建议下一版补充以下内容：

1. 在图一 `ACTION_OVERWRITE` 后补充“按权威域合并”，不要只写保护 20 个字段。
2. 在 `delete旧聚合根→save新数据` 旁标注高风险域例外：支付、退款、库存、沽清、烹饪不参与整单覆盖。
3. 在图二 `ACKNOWLEDGE_CLOUD_VERSION` 后补充冷却期记录和后续动作。
4. 在 `MARK_CONFLICT` 后补充冲突记录结构和处理队列。
5. 在 `mergeRefundDetails` 后补充退款操作号、渠道退款号、金额校验。
6. 在库存扣减失败处补充库存异常表、重试和对账。
7. 在 `postCloudStockGoodsBatch()` 后补充 outbox，避免事务后 HTTP 丢事件。
8. 增加沽清独立同步或权威说明。
9. 增加同步决策日志和 correlationId。
10. 把定时兜底从“覆盖所有变更”改写为“补发未确认操作，快照只做校验”。
