# 通俗描述订单同步

> 基于 `当前版本订单同步机制流程.md` 与任务描述整理。本文只解释当前流程和交互，不替代代码级设计文档。

## 一句话说明

这套订单同步不是简单的“把 POS 的订单复制到云端”，而是让 **POS 本地、云端、支付回调、烹饪设备、库存系统** 一起围绕同一笔订单协作。难点在于：这些系统都可能改订单，但每个字段在不同场景下的“谁说了算”并不一样。

比如：

- 本地 POS 创建订单、取消订单、退货、出餐完成。
- 云端后台可能查询、下发、保存支付和退款数据。
- 在线支付的退款回调先打到云端。
- 烹饪设备通过 IOT-Agent 回调 POS。
- 库存既要本地扣减，也要云端扣减或回滚。

所以它真正解决的是一个问题：**同一笔订单在多个地方被修改后，怎么同步到一致，而且不能把正确数据覆盖掉。**

## 参与者

| 角色 | 通俗理解 | 主要职责 |
| --- | --- | --- |
| POS 本地服务 | 门店机器上的订单系统 | 下单、收银、退货退款、出餐、保存本地 SQLite |
| SQLite | POS 本地数据库 | 保存本地订单、同步状态、版本号 |
| OrderSyncPublisher | 本地同步触发器 | 订单变化后异步推送到云端 |
| 定时任务 | 兜底巡检员 | 每 20 秒或每 2 分钟补偿未同步订单 |
| 云端订单服务 | 中央订单系统 | 接收 POS 推送，保存云端订单，处理云端权威字段 |
| MQTT Broker | 云端到设备的消息通道 | 通知 IOT-Agent 有订单需要同步 |
| IOT-Agent | 门店设备代理 | 接收云端消息，调用 POS 接口，转发烹饪设备事件 |
| 烹饪设备 | 实际执行烹饪的机器 | 接单、排队、烹饪完成后回调 |
| 支付/退款回调 | 在线支付渠道的异步通知 | 在线支付和退款结果通常先进入云端 |

## 两条主链路

当前订单同步可以拆成两条方向相反的链路：

1. **POS 推云端**：POS 本地订单变化后，把订单完整快照推到云端。
2. **云端拉回 POS**：云端订单变化后，通过 MQTT 通知门店，门店 POS 再去云端拉取订单快照。

这两条链路都存在，是因为系统既要支持门店离线/弱网下继续营业，又要支持云端掌握在线支付、退款、后台管理等信息。

## 链路一：POS 推云端

### 1. 本地下单

用户在 POS 上创建订单，比如一笔离线支付订单。

POS 会先把订单保存到本地 SQLite，并标记：

- `localVersion = 1`：本地订单版本号。
- `syncStatus = LOCAL_PENDING`：这笔订单本地有变化，还没成功同步到云端。

保存完成后，POS 会调用 `OrderSyncPublisher.publisher(orderId)`，异步把订单推到云端。

### 2. POS 组装完整订单

推送时，POS 不只是推一个订单主表，而是把订单相关数据一起带上：

- 订单主信息 `OrderModel`
- 商品信息
- SKU 信息
- 商品明细
- 套餐/烹饪工单

可以理解为：POS 把这笔订单的“完整档案袋”发给云端。

### 3. 云端收到后判断怎么处理

云端收到订单后，会先看云端是否已经有这笔订单。

如果云端没有：

- 云端创建订单。
- 保存订单聚合根。
- 尝试扣减云端库存。
- 如果库存扣减失败，当前流程是记录告警，不回滚订单。

如果云端已经有，但内容没变化：

- 云端认为是重复推送。
- 只刷新同步时间。

如果云端已经有，而且内容有变化：

- 云端进入覆盖/合并逻辑。
- 但不会无脑覆盖所有字段。
- 如果云端判断订单已经在线支付，则会保护支付和退款相关字段。
- 如果不是云端支付权威场景，则 POS 的订单数据更可能作为权威数据覆盖云端。

这里的关键点是：**云端现在已经意识到订单不能整体用一个“谁最新谁覆盖”的规则处理，支付和退款字段需要特殊保护。**

### 4. 云端返回 ACK

云端处理后，会返回确认信息，里面带着它确认过的版本。

POS 收到 ACK 后会比较：

- 如果云端确认版本等于本地本次推送版本，POS 把本地订单标记为 `SYNCED`。
- 如果版本不一致，POS 继续保留 `LOCAL_PENDING`，等待下次重试。

这一步的作用是防止“我以为推成功了，其实云端确认的不是我这版数据”。

## 链路二：云端变化拉回 POS

### 1. 云端发出同步通知

当云端有订单需要同步给 POS 时，会调用订单同步通知逻辑，然后通过 MQTT 发一条 `ORDER_SYNC` 消息。

MQTT 不直接改 POS 数据，它只是告诉门店侧：这笔订单云端有变化，你来拉一下。

### 2. IOT-Agent 转发给 POS

IOT-Agent 收到 MQTT 消息后，会异步调用 POS 本地接口：

```text
POST /order/biz-api/pullCloudOrder
```

如果调用失败，IOT-Agent 会记录重试任务，后面继续尝试。

### 3. POS 主动向云端拉订单

POS 收到通知后，不直接相信 MQTT 消息里的数据，而是再向云端发起查询：

```text
GET /pos/orderSync/posOrderPull
```

云端返回完整订单快照，包括：

- 订单主数据
- 商品/SKU/明细
- 支付单
- 退款单
- 烹饪工单

### 4. POS 决定是否应用云端数据

POS 拉到云端订单后，不是立刻覆盖本地订单，而是通过 `OrderSyncDecisionService` 判断。

主要判断逻辑可以通俗理解为：

- 如果不是 POS 本地下的订单，可以直接用云端快照。
- 如果是 POS 本地下的订单，而且本地还有未推送变化，先保护本地，不要让云端覆盖。
- 如果双方版本冲突，而且还在冷却期内，先确认云端版本，暂不乱动。
- 如果冲突过了冷却期，标记为冲突，等待后续处理。
- 如果本地已经同步过，或者云端确实有待应用变化，则应用云端快照。

### 5. 应用云端快照时还要合并字段

对于 POS 本地下的订单，POS 应用云端快照时还有一层合并逻辑：

- POS 业务字段尽量保留本地，例如本地烹饪、交付、门店操作相关字段。
- 云端支付字段按云端权威覆盖，例如在线支付金额、退款状态等。
- 退款明细按 `refundSchId` 合并，同一个退款明细以云端为准，本地独有的退款明细保留。

这说明系统已经进入“字段级权威”的阶段：不是订单整体谁覆盖谁，而是每类字段要有自己的权威来源。

## 兜底机制

实时消息可能丢，HTTP 可能失败，POS 可能离线，所以系统有多层兜底。

### POS 推云端兜底

每 20 秒，POS 会查本地所有 `syncStatus != SYNCED` 的订单，然后继续推云端。

这能覆盖：

- 下单未同步
- 取消未同步
- 退款未同步
- 出餐完成未同步
- 烹饪工单状态未同步

### 云端拉回 POS 兜底

云端到 POS 的方向也有三层兜底：

| 层级 | 周期 | 做什么 |
| --- | --- | --- |
| L1 | 每 20 秒 | 拉本地未同步订单对应的云端数据 |
| L2 | 每 2 分钟 | 按云端最近更新时间窗口拉近期订单 |
| L3 | 较宽窗口 | 查 24 小时内仍处于异常同步状态的订单 |

这几层兜底的目标是：即使 MQTT 丢了，最终也能通过定时任务把订单拉回来。

## 关联业务为什么会放大复杂度

订单同步难，不只是订单字段多，而是订单变化会带动很多外部动作。

### 烹饪工单

POS 创建订单后，会生成烹饪工单，并把状态标记为“发送中”。

流程大概是：

1. POS 创建烹饪工单。
2. POS 推订单到云端。
3. 云端后台看到工单是“发送中”。
4. 云端通过 MQTT 通知门店。
5. IOT-Agent 拉取订单并推送给烹饪设备。
6. 烹饪设备确认接收成功。
7. IOT-Agent 回调 POS。
8. POS 把工单改成“发送成功”和“排队中”。
9. POS 再把这个变化推给云端。

所以烹饪工单不是一次同步就结束，它会随着设备回调继续改变订单。

### 库存

下单时，本地会扣库存，云端也会尝试扣库存。

取消或退款时，本地会回滚库存，并在事务后通知云端回增库存。

这里有两个风险：

- 云端扣库存失败时，当前只记录告警，不回滚订单。
- 本地通知云端库存回增失败时，当前缺少补偿重试和对账任务。

所以库存同步不是“订单保存成功就万事大吉”，它需要额外的补偿和对账。

### 退款明细

退款不是一个简单状态。

比如本地退了商品 A，云端因为在线退款回调记录了商品 B。

如果直接用云端覆盖本地，本地商品 A 的退款会丢。

如果直接用本地覆盖云端，云端商品 B 的在线退款回调会丢。

当前设计按 `refundSchId` 合并：

- 同一个 `refundSchId`：云端覆盖本地。
- 云端有、本地没有：加入本地。
- 本地有、云端没有：保留本地。

这个思路比整单覆盖更安全，因为退款本来就是明细级业务。

## 最容易出问题的场景：在线退款

开发人员担心的核心冲突，可以用一个例子说明。

假设一笔订单是在线支付。

1. 顾客在门店 POS 发起退款。
2. POS 本地先把订单标记为已退货/退款中/已退款，并保存本地退款数据。
3. POS 会把本地变化推到云端。
4. 但在线支付渠道的退款回调不是先到 POS，而是先到云端。
5. 云端收到回调后，也会修改云端订单退款数据。
6. 云端再通过同步链路把退款结果下发给 POS。

冲突就在这里：

- POS 已经认为自己做过一次退款操作。
- 云端也因为支付回调更新了退款结果。
- 两边都不是“无效数据”。
- 但两边修改的是同一笔订单的退款状态、退款金额、退款明细。

如果这时谁覆盖谁没有规则，就可能出现：

- 本地退货状态被云端旧状态覆盖。
- 云端真实退款回调被本地旧快照覆盖。
- 同一笔退款重复执行。
- 退款明细丢失。
- 后续订单管理看到的状态和真实支付渠道状态不一致。

这就是为什么开发人员说“数据权威归属问题就是一个难题”。

## 沽清操作为什么会让问题更麻烦

沽清可以理解为“这个商品卖完了，不能再卖了”。

如果某个订单正在同步、退款、库存回滚过程中，又发生沽清操作，就会出现更复杂的状态组合：

- 订单下单时扣了库存。
- 本地退款时回滚库存。
- 云端可能还没收到退款。
- 云端可能已经执行沽清。
- POS 可能又把旧订单快照推上去。

这时不能只看订单状态，还要看库存状态、商品可售状态、退款状态之间的先后关系。

简单说：**沽清让订单同步从“订单数据一致”升级成“订单、库存、商品售卖状态一起一致”。**

## 当前流程已经做对的部分

从文档看，当前版本已经有几个重要保护：

- POS 推云端时带本地版本，云端 ACK 后 POS 再确认是否同步成功。
- 云端存在订单时，不是完全无脑覆盖，已经保护在线支付/退款字段。
- 云端拉回 POS 时，会根据本地同步状态决定是否应用云端快照。
- 本地有未推送变化时，会优先保留本地 pending 状态。
- 退款明细已经开始按 `refundSchId` 合并。
- MQTT 丢失时，有多层定时任务兜底。
- 取消、出餐完成、烹饪工单发送成功等变化，已经补齐推送触发。

这些说明当前系统不是没有同步机制，而是已经进入了更细的冲突治理阶段。

## 当前流程的核心风险

| 风险 | 通俗解释 | 可能后果 |
| --- | --- | --- |
| 整单覆盖风险 | 订单是一个大对象，但不同字段权威来源不同 | 支付、退款、交付、烹饪数据被覆盖错 |
| 双方同时修改 | POS 和云端都可能改同一笔订单 | 版本冲突、状态回退 |
| 在线退款回调异步 | 真实退款结果先到云端，本地也已操作退款 | 重复退款或退款状态错乱 |
| 库存补偿不足 | 扣减/回滚失败后缺少稳定对账 | 库存和订单不一致 |
| 事件丢失或延迟 | MQTT、HTTP、定时任务都有延迟 | 用户看到旧状态，后续操作基于旧数据 |
| 冲突需要业务判断 | 有些冲突不能靠代码自动判断 | 需要人工介入和审计记录 |

## 讨论时可以抓住的主线

这次讨论不建议只围绕“同步接口怎么调”展开，而应该围绕下面几个问题：

1. **订单哪些字段由 POS 说了算，哪些字段由云端说了算，哪些字段要看场景？**
2. **退款是状态覆盖，还是一组可追踪的退款操作？**
3. **在线退款回调和 POS 本地退款操作如何用同一个业务操作号关联起来？**
4. **库存扣减、库存回滚、沽清之间有没有统一的补偿和对账机制？**
5. **冲突发生后，是自动合并、进入冷却期，还是进入人工处理队列？**
6. **每一次覆盖/保护/合并决策有没有日志，后续能不能查清楚？**

## 一个更好理解的比喻

可以把订单想成一张多人同时编辑的表格。

POS 在改“门店操作列”：下单、出餐、退货、烹饪状态。

云端在改“支付列”：在线支付结果、退款回调、后台订单状态。

库存系统在改“库存列”：扣库存、回滚库存、沽清。

烹饪设备在改“制作列”：发送成功、排队、完成。

如果每个人都把整张表格复制一份，改完后直接覆盖别人的整张表，数据一定会乱。

更合理的方式是：

- 每一列明确负责人。
- 每次修改都带版本号和操作号。
- 不确定谁对时，不覆盖，先标冲突。
- 能自动合并的字段自动合并。
- 涉及钱和库存的操作必须能追踪、能重试、能对账。

这就是当前订单同步真正要解决的问题。
