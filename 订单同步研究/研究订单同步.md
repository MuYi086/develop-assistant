# 研究订单同步

> 研究时间：2026-06-26 | 所属领域：智能餐饮 POS / 云端订单同步 | 研究对象类型：内部业务同步机制

## 一句话定义

当前 POS-云端订单同步机制，是一个“门店 POS 与云端都具备订单修改能力”的离线优先双向同步系统；它的核心矛盾不是网络通信，而是订单字段、业务操作、外部支付回调、库存副作用之间的权威归属和冲突治理。

## 研究边界

本文基于本仓库文档 `当前版本订单同步机制流程.md` 与任务描述展开。

外部资料只作为分布式系统设计参照，不能证明当前项目代码已经具备某项能力。本文所有涉及当前项目的判断，均以源文档为依据；涉及架构建议的部分，属于推演和设计建议。

## 一、纵向分析：复杂性是怎么一步步长出来的

### 阶段一：各端单独闭环，系统看起来并不复杂

开发人员的原话里有一个很关键的信息：**POS 和云端各自都能执行烹饪和支付，各自单独跑都没问题。**

这句话背后的含义是，POS 不是一个纯前端终端，云端也不是一个只做备份的存储端。两边都有业务能力：

- POS 可以在本地下单、收银、退货、退款、触发烹饪。
- 云端可以承载在线支付、退款回调、后台订单管理、库存展示。
- IOT-Agent 和烹饪设备会围绕订单继续产生状态变化。

如果只看单端流程，复杂度是可控的。

POS 单独跑时，它只要相信本地 SQLite；云端单独跑时，它只要相信云端数据库。订单状态从创建到支付、退款、出餐，都是一条线。

问题从“双向同步”开始。

一旦本地 POS 和云端都能修改同一笔订单，系统就不再是单线流程，而是两个时间线在互相追赶。POS 改了一次，云端也可能改了一次；POS 以为自己是最新，云端也可能有更权威的信息。

这时，“订单同步”这四个字容易误导人。它听起来像数据搬运，实际是分布式协作。

### 阶段二：POS 推云端，先解决“本地下单怎么进云端”

当前文档里的第一条主链路，是 POS 到云端的推送同步。

POS 本地下单后，会在 SQLite 中保存订单，并标记：

- `localVersion`：本地版本号。
- `syncStatus = LOCAL_PENDING`：本地有变化，需要推云端。
- `syncedLocalVersion`：已被云端确认过的本地版本。

这个设计的早期目标很明确：门店不能因为网络问题停止营业。POS 先把订单保存下来，再异步推云端。推送失败不要紧，定时任务会继续推。

这就是典型的离线优先取向。

但为了离线可用，系统付出的代价是：云端不能天然成为唯一事实来源。至少在一段时间内，POS 本地拥有一份比云端更新的数据。

当前推送链路已经有几个保护动作：

- POS 推的是完整订单聚合根，不是零散字段。
- 云端会判断创建、重复、覆盖三种动作。
- 云端 ACK 返回确认版本。
- POS 只有在 ACK 版本匹配时才把本地标记为 `SYNCED`。

这说明系统已经在避免“推没推成功说不清”的问题。

但完整聚合根推送也埋下了新问题：**完整快照越大，覆盖风险越高。**

订单里有商品、SKU、支付、退款、烹饪工单、库存关联。如果 POS 每次都推一个完整快照，云端每次都保存一个完整快照，那么只要合并规则不够细，就可能把对方刚刚写入的字段冲掉。

所以文档里出现了 `preserveCloudAuthoritativeFields()`。这说明系统已经从“订单整体同步”走到了“字段级保护”。

### 阶段三：云端拉回 POS，解决“云端变化怎么回门店”

第二条主链路，是云端到 POS 的拉取同步。

云端有订单变化时，通过 MQTT 发 `ORDER_SYNC`，IOT-Agent 收到后调用 POS 本地接口，POS 再主动向云端拉完整订单。

这个设计有一个好处：MQTT 只负责通知，真正的数据以 POS 拉取云端快照为准。这样可以避免 MQTT 消息过大，也能让 POS 在处理时做版本判断。

但它也带来一个现实问题：通知、拉取、应用之间存在时间差。

在这个时间差里，POS 可能又产生了新的本地变化。比如云端通知刚发出，POS 本地又完成了一次退款或出餐。等 POS 拉到云端快照时，云端快照可能不是绝对最新的全局状态。

当前文档里的 `OrderSyncDecisionService` 就是在处理这个问题。

它会根据订单来源、同步状态、版本冲突、冷却期等条件决定：

- 应用云端快照。
- 保留本地 pending。
- 确认云端版本但暂不覆盖。
- 标记冲突。

这一步很重要。它承认了一个事实：**不是每次拉到云端数据都应该立即应用。**

对于 POS 本地下的订单，应用云端快照时还会进入 `mergeSitePosLocalBusinessWithCloudPayment()`，保留 POS 业务字段，同时应用云端支付权威字段。

这进一步说明，当前系统的正确方向不是“统一让谁当主库”，而是“按字段、按业务动作、按场景决定权威”。

### 阶段四：烹饪、库存、退款把同步从数据问题推向业务事务问题

如果订单只有备注、状态、金额几个字段，同步还可以主要靠版本号和字段保护。

但当前餐饮场景里，订单变化会产生外部副作用。

烹饪工单发送成功，不只是订单字段变了，它还代表设备已经接收了任务。

库存扣减成功，不只是 `soldQty` 加了，它还影响后续商品可售数量。

退款回调成功，不只是订单状态变了，它还对应支付渠道真实资金流。

这些副作用不能简单回滚。

Microsoft Azure 的补偿事务模式提到，最终一致性的复杂业务流程发生失败后，不能简单恢复旧值，因为并发实例可能已经修改过数据，补偿逻辑通常必须按业务规则处理。这个观点非常贴近当前订单同步问题：退款、库存、烹饪都不是普通字段覆盖，而是需要业务语义的补偿动作。

当前文档里已经出现了这种复杂性：

- 云端库存扣减失败时，订单不回滚，只记录告警。
- 本地库存回滚后通知云端失败，当前缺少补偿重试/对账。
- 退款明细需要按 `refundSchId` 合并。
- 烹饪工单发送成功后必须再次触发订单重推。

这些不是“同步接口多调几次”能解决的问题。它们需要可追踪的业务操作、幂等键、补偿任务、对账任务和人工处理入口。

### 阶段五：在线退款让字段权威冲突集中爆发

任务描述里最焦虑的点，是在线支付退款。

在线退款的真实链路大概是：

1. POS 本地发起退款。
2. POS 本地先记录退货/退款相关状态。
3. POS 推订单变化到云端。
4. 支付渠道退款结果回调先到云端。
5. 云端修改云端订单退款数据。
6. 云端再把退款结果下发给 POS。

这个链路和离线退款不同。

离线退款里，POS 更容易被视作退款操作事实来源。因为钱款动作可能就在门店完成。

在线退款里，POS 是发起方或业务操作方，但支付结果的事实来源在云端收到的支付渠道回调。POS 本地不能因为自己发起过退款，就覆盖云端回调结果；云端也不能因为自己收到回调，就把 POS 已经发生的退货、库存回滚、商品粒度退款明细全部覆盖掉。

这就是所谓“权威归属需要深入到具体字段”。

更准确地说，权威不只要深入到字段，还要深入到“操作”：

- 谁发起退款？
- 退款请求号是什么？
- 支付渠道退款单号是什么？
- 退款回调对应哪一次退款请求？
- 退货商品明细和资金退款明细是否一一对应？
- 本地库存回滚是否已经执行？
- 云端是否已经确认真实资金流结果？

如果没有这些关联键，只靠订单状态字段同步，在线退款一定会出现难解释的冲突。

## 二、横向分析：几种可选架构路线的取舍

当前系统不是唯一可能的设计。把它放在同类架构方案里横向看，能更清楚地看到它的优势和包袱。

### 方案 A：云端单权威，POS 只是在线终端

这是最容易理解的方案。所有订单修改都必须先请求云端，云端成功后 POS 再展示结果。

优点很明显：

- 订单只有一个权威源。
- 在线支付和退款回调天然在云端闭环。
- 后台管理看到的永远是云端最新状态。
- 冲突大幅减少。

但它和餐饮门店 POS 的现实冲突也很明显：

- 门店网络不稳定时无法可靠营业。
- 烹饪设备和 POS 本地交互会被云端网络延迟拖慢。
- 离线支付、临时断网、设备局域网控制都不好处理。
- 一旦云端不可用，门店核心交易链路受影响。

如果业务要求“门店弱网也能下单、收银、烹饪”，云端单权威就不合适。

### 方案 B：POS 单权威，云端只是备份和查询

这个方案反过来：所有订单事实都以 POS 为准，云端只接收 POS 上报。

优点是门店体验稳定：

- 本地下单、支付、烹饪路径短。
- 网络失败不影响门店继续做生意。
- POS 的本地状态就是操作员看到的状态。

但在线支付会直接打破这个模型。

在线支付和退款回调通常进入云端。支付渠道不会等 POS 在线再告诉它结果。云端收到回调后，如果不能成为支付字段权威，就会出现真实资金流和 POS 订单状态脱节。

所以 POS 单权威只适合纯本地收银或云端支付能力很弱的场景。只要在线支付、云端后台退款、总部管理成为核心能力，POS 单权威就会吃力。

### 方案 C：当前路线，双向快照同步加字段级权威

当前系统更接近这个方案。

它保留 POS 离线营业能力，也承认云端在线支付/退款字段的权威性。两边通过推送、拉取、MQTT、定时任务保持最终一致。

这个方案的优势是现实：

- 兼容门店离线/弱网。
- 兼容在线支付和云端回调。
- 兼容烹饪设备本地事件。
- 可以逐步按字段补齐权威规则。

但它的问题也非常典型：

- 完整快照同步容易误覆盖。
- 字段权威规则会越来越多。
- 业务动作和数据状态混在一起。
- 冲突排查依赖日志和版本设计。
- 退款、库存、沽清等副作用需要额外补偿。

这条路线不是错，但它不能只靠“多写几个 if 保护字段”长期维持。随着场景变多，字段级规则会膨胀成一张隐形大表。团队需要把这张表显式化。

### 方案 D：事件/操作账本为核心，订单快照只是投影

另一条路线是把订单变更改造成事件或操作账本。

不是只保存“订单现在长什么样”，而是记录：

- `OrderCreated`
- `PaymentConfirmed`
- `RefundRequested`
- `RefundCallbackReceived`
- `GoodsReturned`
- `InventoryRolledBack`
- `CookingOrderSent`
- `MealDelivered`

订单当前状态由这些事件投影出来。

Azure 的 Event Sourcing 模式强调，事件以追加方式记录，每个事件描述对象发生的一个逻辑变化，事件存储可以作为权威数据源，并用于审计和历史重建。但它也提醒，这个模式会改变存储、并发、查询、模式演进方式，迁移成本高，不适合无差别套用。

放到当前项目里，事件化最适合的不是整套订单立刻重写，而是优先用于高风险子域：

- 在线支付/退款
- 库存扣减/回滚/沽清
- 烹饪工单状态
- 同步决策日志

这些子域最需要审计、幂等、重放和对账。

### 方案 E：多版本冲突保留，人工或业务规则合并

Apache CouchDB 的复制冲突模型提供了一个值得借鉴的思路：当两个副本离线修改同一文档后再次复制，系统可以保留多个冲突版本，而不是简单丢掉某一边。读取时可以有确定性 winner，但冲突版本仍存在，应用需要合并或让用户处理。

当前订单同步不能直接照搬 CouchDB，因为订单涉及资金、库存、设备副作用，不是普通文档。

但它的思想很有价值：**在不确定谁对时，保留冲突证据比直接覆盖安全。**

当前 `MARK_CONFLICT` 和冷却期设计，已经接近这个方向。下一步应该让冲突不只是一个状态，而是可诊断对象：

- 冲突字段有哪些？
- 本地值、云端值分别是什么？
- 两边版本号和更新时间是什么？
- 涉及哪些支付/退款/库存操作号？
- 系统为什么没有自动合并？
- 后续人工处理动作是什么？

## 三、横纵交汇洞察

### 洞察一：当前复杂度来自一个早期正确决策

允许 POS 本地独立跑通，是餐饮门店系统里的正确决策。

门店不能因为网络抖动停止营业，烹饪设备也不应该每一步都绕云端。这个决策让系统具备离线韧性。

但这个正确决策也决定了今天的复杂度：云端不再是唯一事实源。

当在线支付、退款回调、后台管理继续加强后，系统就从“POS 离线优先”变成了“双主协同”。这不是某个接口写复杂了，而是产品能力叠加后的必然结果。

所以讨论时不应该把问题归咎为“同步逻辑太绕”，而应该承认架构选择的代价：既要门店自治，又要云端权威，就必须有一套明确的冲突治理机制。

### 洞察二：订单不是一个整体，订单是多个权威域的组合

当前最危险的想法，是继续把订单当成一个大对象。

在这个业务里，一笔订单至少拆成这些权威域：

| 权威域 | 更可能的权威来源 | 说明 |
| --- | --- | --- |
| 本地下单基础信息 | POS | 门店现场创建，离线时云端不可见 |
| 在线支付结果 | 云端/支付回调 | 真实资金流结果来自支付渠道 |
| 离线支付结果 | POS | 现金、线下支付由门店确认 |
| 在线退款结果 | 云端/支付回调 | 退款成功失败以渠道回调为准 |
| 退货商品动作 | POS | 门店现场处理商品退回 |
| 库存扣减 | POS + 云端分别执行并对账 | 双库存需要补偿和校准 |
| 库存回滚 | 触发端记录操作，双方最终对账 | 不能只靠快照覆盖 |
| 沽清状态 | 需要定义独立商品权威 | 不宜被订单快照顺手覆盖 |
| 烹饪工单发送状态 | POS/IOT-Agent/设备回调 | 设备事件先回本地 |
| 后台管理状态 | 云端 | 管理侧操作需要下发 POS |

如果这张表不显式存在，代码里就会散落很多“这个字段保留本地、那个字段保留云端”的判断。短期能修，长期难维护。

### 洞察三：退款不应该只被建模成状态字段

在线退款冲突之所以难，是因为它不是状态同步问题。

它至少包含四件事：

1. POS 发起退货/退款业务动作。
2. 本地订单和库存发生变化。
3. 云端或支付渠道执行资金退款。
4. 支付渠道回调确认最终资金结果。

这更像 Saga 分布式事务。Azure 的 Saga 模式把跨服务事务拆成一系列本地事务；每个本地事务完成自己的数据库更新，并通过事件或消息触发下一步，失败时通过补偿事务撤销前面已完成步骤。它也强调复杂工作流更适合编排式 Saga，因为集中控制器能保存状态并处理失败恢复。

当前退款如果继续只看订单快照，很容易丢失“这次退款动作到底走到哪一步”。更稳的做法是把退款建成独立操作流：

- `refundOperationId`：本次退款业务操作号。
- `refundRequestId`：请求支付渠道的退款请求号。
- `paymentRefundId`：支付渠道退款单号。
- `refundSchId`：商品/明细退款关联键。
- `refundStage`：本地退货、请求云端、等待渠道、渠道成功、渠道失败、补偿中、需人工处理。
- `idempotencyKey`：防止重复请求和重复回调处理。

订单的退款状态只是这个操作流的投影。

### 洞察四：覆盖不是最大的问题，无法解释才是

订单错乱后最痛的不是“错了一次”，而是没人能解释为什么错。

一个成熟的同步系统，至少要能回答：

- 本地什么时候改了这个字段？
- 云端什么时候改了这个字段？
- 哪个同步任务把哪个版本推给了谁？
- 哪次合并选择了本地，哪次选择了云端？
- 当时为什么这么选？
- 有没有支付回调、库存回滚、烹饪设备回调参与？

如果这些问题回答不了，后续订单管理就会陷入人工猜测。

所以同步系统需要的不只是 `syncStatus`，还需要同步决策日志。每次应用、跳过、保护、合并、冲突，都应该留下可查询记录。

### 洞察五：沽清和库存不能藏在订单同步后面

当前文档对库存的描述暴露了一个边界问题：订单同步时会触发云端库存扣减，取消/退款时会触发库存回滚，但失败后缺少补偿重试和对账。

库存和沽清是商品售卖域，不只是订单域的附属字段。

订单同步可以告诉库存系统“发生了下单/退货”，但不应该让库存正确性完全依赖订单快照覆盖。尤其是沽清，它可能来自门店操作、后台操作、自动库存规则。如果订单旧快照把商品可售状态带回去覆盖，就会非常危险。

更稳的边界是：

- 订单产生库存操作事件。
- 库存系统记录库存流水。
- 沽清状态有独立权威和版本。
- 订单只引用库存操作结果，不直接成为库存最终真相。

## 四、当前机制风险清单

| 优先级 | 风险 | 触发条件 | 建议处理 |
| --- | --- | --- | --- |
| P0 | 在线退款重复或丢失 | POS 已退款，云端又收到渠道回调 | 用统一退款操作号和幂等键串联 POS、云端、支付渠道 |
| P0 | 云端回调被 POS 旧快照覆盖 | POS 推完整聚合根时支付字段保护不足 | 支付/退款字段建立强权威矩阵，默认禁止旧快照覆盖 |
| P0 | 本地退货状态被云端旧状态覆盖 | 云端下发快照时未识别本地 pending | `LOCAL_PENDING` 场景保持保护，冲突进入诊断队列 |
| P1 | 退款明细合并不完整 | 多商品、多次部分退款、同一商品重复退款 | `refundSchId` 外增加业务操作号、渠道退款号、金额校验 |
| P1 | 库存回滚失败无补偿 | 本地事务成功，通知云端失败 | 建立库存补偿 outbox 与对账任务 |
| P1 | 沽清与退款/库存回滚交叉 | 退款过程中商品被沽清或取消沽清 | 沽清独立版本和权威，不跟随订单快照隐式覆盖 |
| P1 | 冲突状态无法运营处理 | 冲突只记录状态，不记录差异 | 建立冲突详情表和人工处理入口 |
| P2 | 同步链路难排查 | MQTT、HTTP、定时任务多路径并存 | 全链路 `correlationId`、订单版本、操作号日志 |
| P2 | ACK 版本不匹配频繁重试 | 云端确认版本与本地推送版本不一致 | 统计原因，区分正常并发、旧任务、代码缺陷 |

## 五、建议的讨论框架

### 1. 先定字段权威矩阵

讨论前建议先拉一张表，把订单字段按域拆开：

- 订单基础字段
- 支付字段
- 退款字段
- 退款明细字段
- 商品明细字段
- 烹饪工单字段
- 交付/出餐字段
- 库存相关字段
- 沽清相关字段

每个字段至少回答四个问题：

- POS 能不能改？
- 云端能不能改？
- 支付/设备/库存回调能不能改？
- 冲突时谁优先，什么场景需要人工？

没有这张表，代码讨论很容易陷入局部修补。

### 2. 把退款从订单快照里“抬出来”

退款建议按独立业务操作建模。

最低限度需要：

- 退款操作号。
- 幂等键。
- 渠道退款单号。
- 商品退款明细关联键。
- 本地退货动作记录。
- 云端支付回调记录。
- 状态机。
- 补偿和人工处理状态。

订单主表上的退款状态，只作为展示和查询字段。

### 3. 建同步决策日志

每次同步做出以下动作时都记录日志：

- 创建云端订单。
- 云端覆盖 POS 字段。
- 云端保护支付字段。
- POS 应用云端快照。
- POS 保留本地 pending。
- POS 合并退款明细。
- 标记冲突。
- ACK 版本不匹配。

日志字段建议包含：

- `orderId`
- `localVersion`
- `cloudVersion`
- `syncDirection`
- `action`
- `fieldDiff`
- `authorityRule`
- `operationId`
- `correlationId`
- `reason`
- `createdAt`

这类日志不一定给用户看，但必须能支撑开发和运维排查。

### 4. 库存建立 outbox 和对账

库存扣减/回滚不能只靠“HTTP 调一下云端”。

建议本地事务内写入库存操作 outbox：

- 订单退款提交成功。
- 本地库存回滚成功。
- 写入待通知云端库存回滚事件。
- 后台任务可靠重试。
- 云端幂等消费。
- 定时对账本地库存流水和云端库存流水。

这样即使网络失败，也不会丢掉需要补偿的动作。

### 5. 冲突进入可处理队列

不是所有冲突都应该自动合并。

涉及钱、库存、商品明细的冲突，如果规则没有绝对把握，应该进入可处理队列：

- 展示本地值和云端值。
- 展示相关操作记录。
- 展示建议处理动作。
- 允许人工选择保留哪边或执行补偿。
- 处理后写入审计日志。

这和 CouchDB 冲突保留的思想一致：不确定时先保留证据，再合并。

## 六、建议补充的测试场景

| 场景 | 关键断言 |
| --- | --- |
| POS 离线下单后恢复网络 | 云端创建订单，ACK 版本匹配，本地转 `SYNCED` |
| POS 本地取消后 MQTT 下发旧云端快照 | 本地 `LOCAL_PENDING` 不被覆盖 |
| 在线支付订单 POS 发起退款，云端随后收到退款成功回调 | 本地退货动作保留，云端退款结果应用，不重复退款 |
| 同一订单多商品部分退款，本地退 A，云端回调 B | 退款明细按关联键合并，金额不丢 |
| 本地库存回滚成功但云端通知失败 | outbox 保留任务，后续重试成功 |
| 云端库存扣减失败 | 订单不丢失，但生成库存异常对账记录 |
| 烹饪设备发送成功回调后 | POS 更新工单状态并触发重推云端 |
| MQTT 丢失 | 20 秒/2 分钟/24 小时兜底能拉回订单 |
| ACK 版本不匹配 | 本地保持 pending，不误标 synced |
| 冷却期外版本冲突 | 进入冲突队列，并记录字段差异 |

## 七、一个可落地的演进路线

### 近期：先把隐形规则显式化

近期不建议大重构，优先做三件事：

1. 输出字段权威矩阵。
2. 梳理在线退款状态机和幂等键。
3. 增强同步决策日志。

这三件事能直接降低讨论成本和排查成本。

### 中期：把高风险动作事件化

中期可以把支付、退款、库存、烹饪工单这些高风险子域逐步改成操作记录驱动：

- 订单快照继续存在，用于查询和展示。
- 高风险动作必须有操作日志。
- 快照由操作投影或同步合并生成。

这比一次性全量 Event Sourcing 更现实。

### 长期：订单同步从“快照互推”走向“事实流协作”

长期目标不是消灭快照，而是降低快照的权威性。

快照适合展示当前状态，不适合承载所有业务真相。

真正的业务真相应该来自：

- 下单事实
- 支付事实
- 退款事实
- 库存事实
- 烹饪事实
- 同步决策事实

当这些事实都能追踪，订单错乱的概率会下降；即使错了，也能解释、能补偿、能修复。

## 八、讨论会建议问题清单

1. 订单字段是否已有完整权威矩阵？如果没有，谁来补？
2. 在线支付订单的退款结果，是否明确以支付渠道回调为最终资金事实？
3. POS 本地退货动作和云端退款回调之间，当前用什么 ID 关联？
4. `refundSchId` 是否足以覆盖多次部分退款、同商品多次退款、渠道退款单拆分？
5. 云端保护的 20 个支付/退款字段具体有哪些，是否覆盖所有在线退款关键字段？
6. 本地 `LOCAL_PENDING` 时，云端快照里哪些字段仍允许下发覆盖？
7. 冷却期内 ACK 云端版本的业务含义是什么？是否可能掩盖真实冲突？
8. 库存扣减失败只告警不回滚订单，后续由谁处理？
9. 本地库存回滚成功但云端通知失败，是否有可靠重试和对账？
10. 沽清状态由谁权威？订单同步是否可能覆盖沽清结果？
11. 冲突订单有没有后台可视化处理入口？
12. 线上排查时能否看到每次同步的字段差异和权威选择原因？
